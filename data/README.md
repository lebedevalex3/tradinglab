# Data Directory — TradingLab

Этот каталог содержит все данные, используемые в проекте **TradingLab**.  
Структура и правила хранения данных введены намеренно, чтобы обеспечить **воспроизводимость, контроль качества и чистоту репозитория**.

---

## Структура каталогов

```text
data/
  raw/        # исходные данные (как получены от источника)
  clean/      # очищенные и валидированные данные
  features/   # данные с вычисленными признаками
  sample/     # небольшой demo-набор для тестов и примеров
```

## Назначение и правила по каталогам

`data/raw/`

**Что это:**
Исходные данные (OHLCV и т.п.), полученные напрямую от источника (например, биржи через API).

Правила:

❌ НЕ коммитятся в git
❌ Не изменяются вручную

Используются только как вход для очистки и валидации

Назначение:
Источник истины для воспроизводимой обработки данных.

`data/clean/`

**Что это:**
Данные после очистки и базовой валидации:

- исправленные таймстемпы,
- отсортированные бары,
- удалённые дубликаты,
- проверка пропусков и некорректных значений.

Правила:

❌ НЕ коммитятся в git

- Генерируются автоматически из data/raw/
- Используются для ресёрча и бектестов

`data/features/`

**Что это:**
Данные с вычисленными признаками:

- индикаторы (EMA, ATR, Stoch и т.п.),
- нормализованные метрики,
- режимные признаки (volatility, trend, etc.).

Правила:

❌ НЕ коммитятся в git

- Полностью воспроизводимы из data/clean/
- Не считаются “источником истины”

`data/sample/`
**Что это:**
Небольшой демонстрационный набор данных (обычно несколько тысяч свечей).

Правила:

✅ КОММИТИТСЯ в git

Используется для:

- smoke-тестов,
- примеров,
- CI,
- первичной отладки.

Ограничения:

- Должен быть маленьким
- Не используется для реальных выводов о стратегии

Общие правила проекта

В git хранится только каркас и demo-данные

Реальные данные всегда:

- получаются через скрипты,
- описываются в документации,
- могут быть восстановлены повторно

Любой бектест должен указывать:

- какие данные использовались,
- откуда они были получены,
- за какой период.
